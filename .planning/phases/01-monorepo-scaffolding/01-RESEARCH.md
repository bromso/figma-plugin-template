## Summary

Phase 1 establishes the monorepo root: a new `package.json` with Bun workspaces, a `turbo.json` with task declarations, and deletion of the old lockfiles. No code moves in this phase — it is pure scaffolding that later phases depend on.

The existing repo is a flat single-package Vite project with both `package-lock.json` and `pnpm-lock.yaml` present. This phase replaces the root `package.json` entirely, creates `turbo.json`, runs `bun install` to generate `bun.lock`, and deletes the old lockfiles. The directory structure (`apps/`, `packages/`) is created here as empty scaffolding; packages are populated in Phase 2.

At the end of Phase 1, `bun install` must succeed at root, `turbo run build` must recognize the workspace, and no npm or pnpm lockfiles must remain.

**Primary recommendation:** Replace the root `package.json` with the Bun+Turborepo workspace form, create `turbo.json` using the `tasks` key, run `bun install`, and commit `bun.lock`. Do this before touching any source code.

## Architecture Patterns

### Recommended Project Structure (Phase 1 scope — scaffolding only)

```
figma-plugin-template/          ← monorepo root
├── package.json                ← NEW: workspaces, packageManager, private: true
├── turbo.json                  ← NEW: task pipeline
├── bun.lock                    ← NEW: generated by bun install (commit this)
├── apps/                       ← NEW: empty directory placeholder
│   └── .gitkeep
└── packages/                   ← NEW: empty directory placeholder
    └── .gitkeep
```

Files removed in this phase:
- `package-lock.json` — npm lockfile, conflicts with Bun
- `pnpm-lock.yaml` — pnpm lockfile, conflicts with Bun

Files that stay unchanged until Phase 2+:
- All `src/` files, Vite configs, `tsconfig.json`, `figma.manifest.ts`

### Pattern 1: Root package.json for Bun + Turborepo

**What:** Root `package.json` declares workspaces and package manager. All scripts delegate to `turbo run`.

**When to use:** Always — this is the Turborepo + Bun workspace entry point.

**Example:**
```json
{
  "name": "figma-plugin-template",
  "private": true,
  "packageManager": "bun@1.3.11",
  "workspaces": [
    "apps/*",
    "packages/*"
  ],
  "scripts": {
    "build": "turbo run build",
    "dev": "turbo run dev",
    "lint": "turbo run lint",
    "test": "turbo run test",
    "test:watch": "turbo run test:watch"
  },
  "devDependencies": {
    "turbo": "^2.9.5"
  }
}
```

**Critical:** `packageManager` field must use plain semver, no `v` prefix. `"bun@v1.3.11"` will be rejected by Turborepo with a hard error. [VERIFIED: PITFALLS.md — Turborepo regex `(bun|npm|pnpm|yarn)@\d+\.\d+\.\d+`]

**Critical:** `"private": true` is required. Bun workspaces require the root package to be private. [VERIFIED: Bun workspaces official docs]

### Pattern 2: turbo.json with `tasks` key

**What:** Declares the task pipeline. Uses `tasks` (not deprecated `pipeline` key removed in Turborepo 3.0).

**When to use:** Required before any `turbo run` command will function.

**Example:**
```json
{
  "$schema": "https://turbo.build/schema.json",
  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": ["dist/**"]
    },
    "dev": {
      "cache": false,
      "persistent": true
    },
    "lint": {
      "outputs": []
    },
    "test": {
      "dependsOn": ["^build"],
      "outputs": ["coverage/**"]
    },
    "test:watch": {
      "cache": false,
      "persistent": true
    }
  }
}
```

**Notes:**
- `"dependsOn": ["^build"]` for `build` and `test` enforces topological ordering (BUILD-02)
- `"cache": false, "persistent": true` for `dev` and `test:watch` — these tasks never exit; caching them would cause Turborepo to hang
- `"outputs": ["dist/**"]` enables Turborepo to cache and restore build artifacts
- `tasks` key is required in Turborepo 2.x; `pipeline` key was deprecated in 2.0 and removed in 3.0 [VERIFIED: STACK.md sourced from Turborepo release docs]

### Pattern 3: bun.lock as committed text file

**What:** After `bun install`, commit `bun.lock` (text format, not the old binary `bun.lockb`).

**When to use:** Always — the text lockfile enables deterministic installs and is human-readable.

**Notes:**
- Bun 1.1+ generates `bun.lock` text format, not binary `bun.lockb`
- `bun.lock` must be committed to the repository for reproducible installs
- Do NOT add `bun.lock` to `.gitignore`
- Turborepo parses `bun.lock` for dependency change detection (cache invalidation) [VERIFIED: PITFALLS.md — Turborepo lockfile parsing documented]

### Anti-Patterns to Avoid

- **`pipeline` key in turbo.json:** Deprecated in Turborepo 2.x, removed in 3.x. Always use `tasks`.
- **`"packageManager": "bun@v1.3.11"` (v prefix):** Turborepo rejects it. Use `"bun@1.3.11"`.
- **Keeping both `package-lock.json` and `bun.lock`:** Having both lockfiles causes Turborepo and Bun to conflict on which lockfile is authoritative. Delete npm/pnpm lockfiles in this phase.
- **`"private": false` on root package:** Bun refuses to process workspaces without `"private": true`.
- **Running `turbo` before setting `packageManager`:** Turborepo emits warnings and may fall back to incorrect package manager detection. Set `packageManager` before any `turbo` invocation.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Topological build ordering | Custom `prebuild` + `npm-run-all` chains | `turbo run build` with `"dependsOn": ["^build"]` | Manual ordering breaks when packages are added; Turborepo derives order from the dependency graph automatically |
| Workspace package linking | Symlinks or relative `file:` paths | `workspace:*` protocol in Bun | Bun's workspace protocol handles symlinking, hoisting, and resolution correctly; `file:` paths do not follow updates automatically |
| Parallel script execution | `npm-run-all`, `concurrently` | Turborepo `persistent: true` tasks + `bun run --parallel` | Turborepo manages concurrency with awareness of task dependencies; ad-hoc parallelism doesn't know about the task graph |

**Key insight:** Turborepo's value is cache-aware, dependency-ordered task execution. Once `turbo.json` is correct, `turbo run build` replaces all manual script chains.

## Common Pitfalls

### Pitfall 1: `packageManager` Field Rejects `v` Prefix

**What goes wrong:** `"bun@v1.3.11"` causes Turborepo to fail immediately with `could not parse the packageManager field in package.json`.

**Why it happens:** Turborepo validates against `(bun|npm|pnpm|yarn)@\d+\.\d+\.\d+`. The `v` prefix does not match.

**How to avoid:** Use `bun --version` to get the exact string (`1.3.11`), then write `"bun@1.3.11"` — no prefix.

**Warning signs:** `turbo build` fails before any task runs with `invalid_package_manager_field`.

### Pitfall 2: Old Lockfiles Cause Conflicting Resolution

**What goes wrong:** Bun detects `package-lock.json` or `pnpm-lock.yaml` and may warn or behave unexpectedly. Turborepo may read the wrong lockfile for cache invalidation.

**Why it happens:** Multiple lockfiles in the same repo create ambiguity about which dependency tree is authoritative.

**How to avoid:** Delete both `package-lock.json` and `pnpm-lock.yaml` before running `bun install`. Commit `bun.lock` immediately.

**Warning signs:** `bun install` succeeds but prints warnings about detected foreign lockfiles; Turborepo cache invalidates unexpectedly.

### Pitfall 3: `bun.lock` Not Committed

**What goes wrong:** Without `bun.lock` committed, different machines get different dependency trees. Turborepo cache becomes unreliable across environments.

**Why it happens:** Developers may assume lockfiles are gitignored (npm convention sometimes does this for libraries).

**How to avoid:** Explicitly add `bun.lock` to the repository. Verify it is not in `.gitignore`.

**Warning signs:** `bun install --frozen-lockfile` fails on CI; team members see different resolved versions.

### Pitfall 4: Empty `apps/` and `packages/` Directories Not Tracked by Git

**What goes wrong:** Git does not track empty directories. If `apps/` and `packages/` contain no files, they disappear from the repo and Bun workspace globs fail on a fresh clone.

**Why it happens:** Standard Git behavior — empty directories are not tracked.

**How to avoid:** Add a `.gitkeep` file inside each directory, or create the first child package stub immediately (Phase 2 will populate them anyway, but the scaffolding phase should ensure the directories exist).

**Warning signs:** Fresh `git clone` followed by `bun install` prints "No workspace packages found".

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `turbo.json` `pipeline` key | `turbo.json` `tasks` key | Turborepo 2.0 (2024) | `pipeline` is silently ignored in 3.x; always use `tasks` |
| `bun.lockb` binary lockfile | `bun.lock` text lockfile | Bun 1.1 (late 2024) | Text format is diffable, readable, and the only format going forward |
| npm/pnpm workspace with `workspaces` in `package.json` | Bun workspace with same `workspaces` field | Bun 1.0 (2023) | Bun uses the same field format as npm — no migration needed, just swap the package manager |

**Deprecated/outdated:**
- `pipeline` key: Turborepo 2.0 deprecated it; 3.0 removed it silently
- `bun.lockb`: Binary format; Bun 1.1+ writes text `bun.lock` instead
