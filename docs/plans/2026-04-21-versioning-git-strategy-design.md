# Design: Versioning, Git & Branch Strategy

**Date:** 2026-04-21
**Status:** Approved

## Problem

The template has no formal versioning — no version fields in package.json, no CHANGELOG, no automated release process. Git tags (v1.0-v1.2) were created manually. Contributors (human and AI) have no documented branching or commit conventions.

## Solution

### 1. Changesets
Install `@changesets/cli`. Contributors run `bun changeset` with each PR to describe what changed. Changesets accumulate, then a "Version Packages" PR auto-bumps versions and generates CHANGELOG.md.

- Root package versioned as `figma-plugin-template` (tracks template releases)
- Internal packages (`@repo/ui`, `@repo/common`) versioned together
- `@repo/storybook` ignored (documentation, not a versioned artifact)
- `access: "restricted"` (private repo, no npm publish)

### 2. GitHub Actions Release Pipeline
New `release.yml` workflow:
- On push to master: runs `changeset version`, opens "Version Packages" PR
- On merge of version PR: creates GitHub Release with tag + changelog

### 3. GitHub Flow Branching
- `master` = always releasable, protected
- Feature branches: `feat/`, `fix/`, `chore/`, `docs/`
- All changes via PRs, must pass CI
- Squash merge preferred

### 4. AI Guidance
CLAUDE.md updated with versioning instructions. `build-figma-plugin` skill includes changeset step.

### 5. Version Fields
- Root: `"version": "1.3.0"` (current milestone)
- Internal packages: `"version": "0.0.0"` (changesets manages from here)
